LINKEDIN GTM DATA · SYSTEM OVERVIEW

Corporate Pain-Point Dashboard → LinkedIn Ads Targeting

A full-stack tool that ingests LinkedIn Sales Navigator scrape bundles, extracts corporate pain-points with Claude, clusters them into synthesized insights, and exports audience CSVs ready for upload to LinkedIn Campaign Manager — turning unstructured social-media chatter into precisely-targeted ad audiences.

Node 22 TypeScript Express PostgreSQL React 19 + Vite Tailwind Haiku 4.5 Opus 4.7 Railway Vercel

1. What this is

The standard LinkedIn Ads workflow is blind: pick a company size + seniority + industry from Campaign Manager, write copy, pray the targeting lands. This tool replaces the blind part — it reads what decision-makers are actually posting about in a specific industry, extracts the pain-points they're expressing, and produces an audience list of people whose companies are publicly talking about those pains.

Three artifacts come out of the pipeline, in order of business value:

  • A pain-point catalog per micro-segment — 10–20 named themes, each backed by verbatim quotes.
  • A contact list of decision-makers at companies that posted about each theme, filterable by seniority / industry / recency / connection-degree.
  • A LinkedIn Matched Audiences CSV in the exact format Campaign Manager accepts — first/last/company/title + a URN-form profile URL that LinkedIn canonicalizes to the member.

Current live cohort: uk-industrial-iot::ex-london — 491 companies, 1,151 contacts, 5,725 analyzed posts, 15 synthesized insights, ~4,500 post→insight tag edges.

Why a join table for tags, not a JSONB array

The hot query for the audience builder is "contacts at companies whose posts tag into insight X" — that's a relational predicate on a 24-row dimension (the insights menu). A B-tree index on insight_id handles it in microseconds. GIN on a JSONB array would work but adds complexity and doesn't support per-edge metadata.

The source + score columns are what make the three tagging passes coexist: evidence-bootstrapped rows keep their ground-truth 1.0 score; fuzzy rows carry their similarity score for later threshold tightening; LLM rows carry the model's reported confidence. One unique index per post-kind makes every pass idempotent — re-running is safe.

4. Ingest layer

LinkedIn Sales Navigator is a paid search product. Exporting its HTML is the primary data source because it bundles three things at once that would otherwise each cost API credits: company profile data, decision-maker cards, and recent company-page posts.

Key discovery
Sales Nav company pages embed up to ~20 of the company's recent LinkedIn posts inline, as [data-anonymize="general-blurb"] elements with the full post text in the title attribute plus a <time datetime> ISO timestamp. Parsing these yields 6,121 free posts for the IoT cohort — zero scrape API calls, zero credits. Replaced the original plan of paying ScrapeCreators per company.

Extraction mechanics

// src/parsers/salesnav-html.ts — post extraction

$("article.alert-card").each((_, el) => {
  const alertId = el.attribs["data-alert-id"];
  // urn:li:fs_salesEntityAlert:(urn:li:fs_salesCompany:<co>,<postId>,TYPE)
  const [, urnCo, postId] = alertId.match(/fs_salesCompany:(\d+),(\d+),/);
  const blurb = $(el).find('[data-anonymize="general-blurb"]').first();
  const post_text = blurb.attr("title") ?? blurb.text();  // full text, not DOM-clamped
  const post_date = $(el).find("time[datetime]").attr("datetime");
  // Synthetic URL so (post_url) stays UNIQUE even before ScrapeCreators fills in.
  const post_url = `salesnav://company/${urnCo}/alert/${postId}`;
  …
});

What the seeder does

scripts/seed-from-linkedin-salesnav.ts is a single idempotent pass:

  1. Loads the three CSVs + every HTML file into memory as maps keyed by linkedin_company_id.
  2. For each id present in either Sales Nav CSV or HTML bundle, builds a merged company record (Sales Nav metadata wins over CSV; HTML details win over either).
  3. Upserts companies via linkedin_company_id (primary key) or primary_domain (fallback) — never duplicates.
  4. For each parsed decision-maker, derives role_seniority + public_linkedin_url inline and upserts contacts.
  5. For each embedded post, inserts a scraped_company_posts row — ON CONFLICT (post_url) DO NOTHING.

5. Analysis layer

Each company-page post is run through Haiku 4.5 once. The model extracts a structured summary; the row is persisted in company_post_analysis with a 1:1 relationship to the source post. Idempotent (ON CONFLICT DO NOTHING).

Per-post extraction prompt shape

Input (per post)

  • post_text (clamped to 2,000 chars)
  • authorName — company display name
  • authorTitle — "Company Page"
  • authorCompany — company display name
  • postDate (ISO)
  • microsegmentLabel — for context

Output (strict JSON)

  • topics[] — 3–10 short strings
  • pain_points[] — 0–10 short strings
  • sentiment — positive / neutral / negative / mixed
  • sentiment_score — -1..1 float
  • intent_signals[] — e.g. thought_leadership, evaluating_vendors
  • key_quotes[] — verbatim substrings of the post

Why Haiku, not Sonnet/Opus

Constraint
The project runs on an ANTHROPIC_AUTH_TOKEN (OAuth bearer from claude setup-token), which is tier-gated to Haiku for programmatic use. Sonnet 4.6 and Opus 4.7 return instant 429s on that auth class regardless of volume — it's a subscription gate, not throughput throttling (see §14 Key decisions). Haiku's per-post extraction quality is already strong on this data. When Sonnet is wanted anyway, it's reached via Claude Code subagents (below).

6. Synthesis → the insight catalog

Per-post extractions are the raw material; they aren't yet the product. The synthesis step clusters them into 10–15 named pain-point themes per microsegment, each with a handful of ground-truth evidence posts. This is the catalog the ads operator actually uses.

The Opus-via-subagent workaround

Because the synthesis call benefits from stronger reasoning, and the OAuth token can't reach Opus directly, synthesis is run through a Claude Code subagent with model: "opus". The subagent itself is a Claude instance authenticated under the Claude Code session (which is entitled to Opus), so the round-trip works: 

// 1. Dump the ~200 most informative post extractions to disk.
npx tsx scripts/export-synth-input.ts
    → writes scripts/_synth-input.json  (capped ~250 KB)

// 2. Spawn an Opus subagent. It reads the JSON, applies the
//    pain_synthesis prompt, and writes 10–15 insights to disk.
Agent({ model: "opus", prompt: "Read _synth-input.json, apply prompt,
                                 write _synth-output.json" })

// 3. Import the output — wipes the old cohort insights (cascading
//    delete to post_insight_tags) and inserts the new ones.
npx tsx scripts/import-synth-output.ts

Haiku is also a valid synthesis model (with max_tokens: 16384 and a retry loop on 429/529s) — src/services/post-analysis.ts uses it directly when analyze --stage 2 is invoked. The Opus subagent path produces sharper, more ad-ready insight names.

7. Tagging system (connects posts ↔ insights)

Without this layer, insights are read-only summaries. Tags are the bridge that lets the audience builder ask "give me contacts at companies that posted about X". Three-pass build, all idempotent, all coexist in the same table via a source column.

pass 1 bootstrap-tags-from-evidence.ts

Evidence bootstrap

Synthesis already hand-picked 3–8 posts per insight as evidence (in pain_point_insights.evidence). That's ground truth — mirror each {post_url → insight_id} edge into post_insight_tags with source='evidence', score=1.0. Cost: zero. Typical yield: ~80 rows.

pass 2 run-fuzzy-tagging.ts

Fuzzy (pg_trgm)

Explode each post's pain_points[] + topics[] and each insight's insight_name + topics[] into tokens, compute pairwise similarity(), take the max per (post, insight), insert where ≥ 0.55. Pure SQL; no API cost. Yield: ~1,400 rows.

pass 3 run-llm-tagging.ts

LLM residual

For posts still untagged after passes 1 + 2, send a menu-classification prompt to Haiku: "Given this post and these 15 insight options, which apply?" Returns {matches:[{insight_id, confidence}]}. Filter confidence ≥ 0.4. Yield on IoT: ~3,060 rows.

Why not just LLM everything
At ~$0.80/MTok (Haiku 4.5), LLM-tagging 5,725 posts costs ~$2 per run. Fuzzy is free and catches the lexically-obvious matches (the majority). Running fuzzy first cuts LLM volume by ~40%, tightening the budget and cutting tagging time ~60%. The three sources cohabit the table, so future iterations can re-weight (e.g. "only evidence tags for conservative audiences").

8. Public URL resolution

LinkedIn Matched Audiences canonicalizes by member URN, not vanity slug. Sales Nav URLs embed the URN in the first comma-separated segment; you can rebuild a LinkedIn-accepted URL entirely client-side, no API calls needed.

The trick

// Sales Nav URL (what we scraped):
//   https://www.linkedin.com/sales/lead/ACwAAARld…_RM,NAME_SEARCH,kF-Q
//                                         └── LinkedIn member URN ──┘
//
// Rebuilt URL (what LinkedIn accepts):
//   https://www.linkedin.com/in/ACwAAARld…_RM
//
// LinkedIn 301-redirects /in/<URN> → /in/<vanity>.
// Matched Audiences hashes the URN directly — so URN-form URLs match
// the member graph the same as vanity URLs.

const m = salesnavUrl.match(/\/sales\/lead\/([^,?\s/#]+)/);
const publicUrl = m ? `https://www.linkedin.com/in/${m[1]}` : null;

scripts/rebuild-public-urls.ts runs this regex across all contacts and writes contacts.public_linkedin_url. The seeder calls it inline on every insert. Audience exports prefer this column over salesnav_lead_url, so the CSV's linkedin_url cells are LinkedIn-matchable out of the box.

Why this matters
Alternatives cost money (Proxycurl — ~$0.05/lookup × 1,151 = $60), require new vendor accounts, and add PII handling. This regex replaced the entire vendor-swap migration plan.

Audience Builder (Contacts tab)

The operator's primary surface. Works like a funnel:

  1. Jump in via "Build audience →" on any insight card (preselects that insight).
  2. Multi-select chips: pain-points, industries, seniority, connection-degree.
  3. Single-select chips: recency window (30d / 90d / 365d / all-time).
  4. Live counter updates on every click (/audiences/preview).
  5. "Export CSV →" streams the full list to disk, ready for LinkedIn upload.

Each contact row shows name, title, company, location, connection-degree, role-seniority pill, expandable bio, and a LinkedIn ↗ link that opens the URN-form public URL.

11. End-to-end LinkedIn Ads workflow

  1. Scout. Browse insights for the cohort, read evidence posts, pick the pain-point that matches your pitch (e.g. Reactive maintenance blind spots).
  2. Narrow. Click "Build audience", layer on seniority (usually CEO/Founder/MD + C-Suite for decision-maker campaigns), industry focus, and recency (last 90 days if you want fresh signal).
  3. Size. Watch the live counter. LinkedIn's minimum audience is ~300; practical campaign targets are 500–5,000. Loosen or tighten filters until you're in range.
  4. Export. Click "Export CSV →", verify the row count matches the counter, open the file, confirm linkedin_url cells all look like linkedin.com/in/<URN>.
  5. Upload. LinkedIn Campaign Manager → Audiences → Create → List upload → Contact list → paste the CSV. Expect 60–90% match rate (LinkedIn canonicalizes URNs to members).
  6. Copy. Pull ad hooks from the insight's key_quotes array and pain_point_summary — those are verbatim words from the target audience, which is copy gold.
Why this beats standard LinkedIn targeting
Standard LinkedIn Ads lets you target by job title + industry + company size. This tool adds a layer Campaign Manager can't: "has personally been exposed to pain-point X recently" — inferred from their company's own LinkedIn posts. A CFO at a company whose last three posts complained about unplanned downtime is a warmer lead than a random CFO at an industrial manufacturer, even if both look identical on LinkedIn's own filters.

14. Key design decisions

Primary data source: Sales Nav HTML, not scraping APIs

Sales Nav bundles include company profile + decision-makers + recent company posts in one scrape. Parsing them yielded 6,121 free posts and 1,151 enriched contacts for the IoT cohort — an entire analysis cohort sourced for zero API credits. We still wire ScrapeCreators for the case where no HTML bundle exists, but it's now a fallback, not the main path.

Public URL resolution: regex, not vendor

LinkedIn Matched Audiences accepts /in/<URN> URLs; LinkedIn itself 301s them to the vanity. The URN is embedded in every Sales Nav lead URL. One regex extracts it. We avoided Proxycurl ($60) and Apollo (PII handling) entirely.

Tagging: join table + fuzzy + LLM hybrid

Join table over JSONB array because the hot query is a relational predicate on a 24-row dimension. Fuzzy first because most matches are lexically obvious and SQL is free. LLM residual only because the marginal 30% of posts need semantic matching. All three sources live in the same table, scored + annotated.

OAuth token tier-gate — Opus via subagent

sk-ant-oat01-… tokens from claude setup-token are Haiku-only for programmatic use. A developer key (sk-ant-api03-…) would unlock Sonnet/Opus but requires separate billing. Workaround: use Claude Code subagents with model: "opus". They run under the Claude Code session's entitlement, so synthesis-via-Opus is free under the existing subscription.

Role bucketing: shared TS, not SQL CASE

Role-seniority used to live as a SQL CASE in /contact-stats AND as a JS function on the client. Extracted to src/lib/role-bucket.ts — canonical, called from the backfill script, seeder, and dashboard. Column is materialised so filters use an index, not a per-row LIKE.

Synthetic post URLs for HTML-derived posts

scraped_company_posts.post_url is UNIQUE, but Sales Nav HTML doesn't expose the public LinkedIn post URL. Fabricated a stable salesnav://company/<co>/alert/<id> scheme so re-seeds are idempotent without collision risk, and real HTTPS URLs coexist when ScrapeCreators later fills them in.

15. Known limits + roadmap

Limits right now

  • One cohort is analyzed. The IoT microsegment is fully processed. The other 39 microsegments have contacts but no posts / insights / tags — they need HTML bundles.
  • No emails. LinkedIn Matched Audiences accepts the URN-URL form we generate, but email enrichment (via Apollo or similar) typically improves match rate further. Optional future step.
  • No ad-copy generator. The insights contain the raw material (key_quotes, who_feels_pain, what_triggers_it) but nothing automatically produces headline/body variants. Manual copy lift for now.
  • No temporal trend per insight. We have post_date on every tag, but nothing visualises whether pain-point X is gaining or fading over time. Would help budget allocation.
  • Single-user auth model. No login screen; whoever has the Vercel URL can see everything. Fine for a single team; add OAuth before customer-facing exposure.
  • No saved audiences. Every export is generated on the fly. To rebuild the same audience after new data arrives, re-enter the filter state — there's no named-audience store.

Natural next additions

  • Saved audiences — audiences table keyed on filter state; one button to rebuild and re-export.
  • Ad-copy generator — Opus subagent takes insight + top 6 quotes + seniority mix → 3 headlines (≤70 chars), 3 body variants (≤150), 1 CTA, angle classification.
  • Account rollup — companies with ≥N tagged posts become a Matched Company Audience seed (different CSV shape).
  • Temporal trend — sparkline per insight card showing 30/90/365-day tag counts. Falling pains burn ad budget; rising pains deserve it.
  • Lookalike expansion — from a small audience, find untagged contacts whose (industry, seniority, location, size) vector matches. Purely SQL k-NN.
  • Email enrichment — backfill contacts.email via Apollo/Clearbit; raises LinkedIn match rate.
